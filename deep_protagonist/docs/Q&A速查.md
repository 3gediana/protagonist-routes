# Q&A 速查

> 用户问过的问题 → 答案 + 上下文。

## Q-001 · "为什么这么小的参数模型要训练这么久？"

**核心答案**：RL 的本质就是慢。模型大小不是瓶颈，**agent 必须自己在世界里跑出数据**才是瓶颈。

类比：
- 大模型（GPT）= 学生在图书馆翻书，几秒一本
- RL = 学生必须自己出门走一步看一步，撞墙才知道是墙

3-8 周里 80% 时间 CPU 在算环境，GPU 在等。

## Q-002 · "不牺牲效果有什么加速办法？"

4 招（**S0/S1 阶段架构里默认全开**）：

1. **并行环境**：同时开 28 个世界，agent 在 28 个平行宇宙试错（加速 4-8 倍）
2. **C++ 环境优化**：纯 C++ 写得极快（2-3 倍）
3. **混合精度**：FP16 算神经网络（1.5-2 倍）
4. **课程学习**：先简单再难，agent 不迷失（2-5 倍）

加起来 3-8 周 → 5-12 天。会牺牲效果的招（砍模型/简化环境/跳过课程）**不用**。

## Q-003 · "训练可以中断和续训吗？"

可以，叫 **checkpoint + resume**：
- 每 5 分钟或 10 万步自动存档
- Ctrl+C 会触发立即存档
- 启动时自动找最新存档接着跑
- 占空间 200-500MB / 份，滚动保留最近 10 份 + 历史最好 3 份

## Q-006 · "3D 升级成本如何？"

## Q-008 · "什么是 PPO？"

PPO（**P**roximal **P**olicy **O**ptimization）= 近端策略优化。简单说：

- agent 有一个"打法"（神经网络）
- 训练时让它在世界里玩一段（比如 5 分钟），收集"做了啥 / 得了多少分"
- 用这段数据调整网络旋钮，**但每次只允许小步调整**——这就是"近端"二字
- 防止一次跨太大学坏（比如训练崩盘）
- 反复几千万次循环，agent 越来越会玩

类比：
- 教小孩骑车 → 不能让他每次摔大跤就完全推翻所有动作。要慢慢调每个细节
- 这就是 PPO 的精神

## Q-009 · "S3.7 做的这一堆系统是干啥用的？"

为 **S4 PPO 训练**做准备。PPO 训练前必须有：

1. **AgentAction**（动作接口）：神经网络输出的动作格式 → S3.7a ✅
2. **RewardSystem**（奖励函数）：每帧给 agent 一个分数 → S3.7b ✅
3. **EpisodeManager**（一局边界）：何时算输 / 何时重新开始 → S3.7c ✅
4. **Observation**（观察向量）：神经网络的输入 → S3.7d ✅
5. **完整的世界机制**：资源、背包、庇护所、夜行狼 → S3.7e/f/g/h ✅

少任何一个 PPO 都跑不起来。

## Q-010 · "为什么背包只有 3 格？这么小？"

蓝图原文 line 45："**持有物品：0-3 件简单物品**"。这是用户当初拍板的。

- 太多格 = agent 学不会取舍，无脑捡所有东西
- 3 格 = 必须做选择（要带石头还是干柴？带不带长矛？）
- 这本身就是 RL 训练的"难点设计"

## Q-011 · "庇护所建造为什么只要 1 木 + 1 草？"

最初设计是 2W+2G（共 4 件），但背包只能 3 格——agent 永远凑不齐 4 件材料一次性建造。

两条路：
- 加 stockpile（材料堆）系统，复杂度高
- **降到 1+1（共 2 件，一次装下）**，简洁

选了后者。蓝图没规定具体材料数量，只说"用树枝草秆搭简易窝"。

- **129**：建好房子数 / 5 (0-1)

## Q-013 · "为啥不一开始就让 agent 啥都能做？非要分阶段？"

因为 RL 训练有个大坑叫**稀疏奖励**：

- 如果一开始放开所有动作（30+ 个），agent 随机乱试 → 几乎不可能凑巧建好一座房
- 没成功就没奖励 → 不知道怎么改 → 学不到任何东西

**主线分阶段**就是把"难题"切成几个"小题"：
- 早期：只学采草采果（任务简单，10 分钟就有正反馈）
- 中期：能采集了再学建造（前置技能已掌握）
- 后期：有房住了再学农耕（更多前置技能）

每阶段完成给 **+50 大奖励**，引导 PPO "为了过关而努力"。这就是**课程学习**。

## Q-014 · "建房子的'图纸'怎么工作？"

像盖建筑工地：

1. 你 P 键放下"图纸"——地上画个白框，标着"需要 4 木+2 草"
2. 你去采木头、采草，背包里装着
3. 走到图纸边，R 键放材料进去（不一定 4+2 一次放完，可以分多次）
4. 系统每帧检查"够没够"
5. 够了，**白框变成 3D 真房子**（凭空盖好）

蓝图原话："只要收集材料放到那边，系统判断材料够了就自动建好 3D"——完全照做。

## Q-015 · "驯服的'信任度'怎么涨？"

- K 键喂食：rabbit/deer 吃 1 个草 → trust +0.10；wolf 吃 1 个兽皮 → trust +0.15
- 累积到 **0.5** 触发"follower" 状态
- 之后该动物就自动跟着你跑，不再自由乱走
- 蓝图原话"长期喂食 → 建立 trust → 跟随"

## Q-019 · "并行项目用 DNC 我们要不要用？"（2026-05-24 14:00）

**不用**，文献+对比给了 5 条结论：

1. **DNC 训练困难、不稳定**：Stanford / 瑞典 DiVA / PMC 论文都明确说 "difficult to train, prone to overfitting, slow convergence, complex memory operations"。需要 curriculum learning + bi-directional + transfer learning 才能稳

2. **业界已经放弃 PPO + DNC**：2024 Memory Gym (ICLR)、2025 RLBenchNet 大规模对比 MLP/LSTM/GRU/Transformer-XL/Mamba **完全没列 DNC**——主流不在选项里

3. **DNC 用于导航需要绝对坐标**（Parisotto-Salakhutdinov 2018），我们 obs 没有

4. **DNC 真正擅长的是图谱推理**（伦敦地铁、家谱、bAbI QA），不是实时控制

5. **场景不一样**：protagonist_ecology 是多 agent + 社交记忆 + 跨 episode 知识，DNC 甜点；我们是单 agent + 5 分钟 episode + 空间记忆，GRU + Spatial Memory 甜点

我们选了 **GRU(256) + SpatialMemory(64×64×6) + obs P0 补料**（Parisotto-Salakhutdinov 2017 "Neural Map" 路线），见 D-037。

## Q-020 · "GRU 只能记几秒，要长期记忆怎么办？"（2026-05-24 14:00）

四层叠加方案（一起用）：

| 层 | 谁负责 | 时间窗 |
|---|---|---|
| **当前 obs** | encoder MLP | 这一帧 |
| **GRU(256)** | 短期记忆 | 几秒-十几秒 |
| **SpatialMemory 64×64×6** | 一局空间地图 | 整局 5 分钟 |

## Q-018 · "PPO 第一次 1M 训练，怎么 ep39 后 reward 暴涨到 +6000？"（2026-05-24 13:25）

**经典 reward hacking 案例**——PPO 找到了奖励函数的 bug 在疯狂刷分，不是真的学到了什么。

具体经过：
- 蓝图给 drink 的奖励权重 = `0.03 × ml`，每次按 Q 喝 35ml = +1.05 reward
- VitalSystem.drink 在 thirst 满 100 时已经"装不下"了，actual delta 为 0
- 但 RewardSystem 不知道，照样按"试图喝的量"给奖励
- 9000 ticks × +1.05 = 上限 +9450 / episode

PPO 在 ep39 偶然探索到河岸 + 反复按 Q 的组合，立刻锁死这条路径。然后所有 episode 都开始疯狂刷水。死亡率为 0（站着不动当然不死），但 milestone、building、kills 全部归零——agent 永远不会去探索建造和农耕。

**修复**（D-036）：让 `vitals.drink/eat` 返回 actual_delta，调用方传给 `reward.event_X(actual)`。这样喝不进去的水不给 reward。`runs/ppo_run1_buggy.jsonl` 留作教学样本。

**RL 工程师必须记住的铁律**："奖励应该等于 agent 实际获得的，不是试图获得的"。所有 reward 函数设计前先问自己：这个量有上限吗？上限到了还会触发 reward 吗？如果会，就是 reward hacking 漏洞。

## Q-021 · "PPO 跑了 1M 步 mean +18 卡住，是不是网络太小？该扩到 5M 参数吗？"（2026-05-24 15:20）

**不一定**。决策 D-039 之前我也以为是参数不够，差点盲投 20× 算力（HIDDEN 512 + 5M 步）。先做了 30 分钟"行为直方图"诊断救了这一脚。

**诊断方法**（`scripts/analyze_actions.py`）：
- main_train.cpp 给每个 episode 写 act:[16 个动作触发次数] 进 jsonl
- 用 ppo_run3 checkpoint 跑 5 局 eval，看每个动作的触发频率
- 同时算 mean |move| 和 mean |yaw_rate|

**当时看到的真相**：
- 16 个离散动作触发率 35-73%，**每步同时按多个键**
- mean |move|=0.9（一直满速跑）/ |yaw|=1.7 rad/s（一直疯转）
- r_water 看起来稳 +1.8 是因为"站河边乱按必中"，不是学会找水
- r_craft / r_milestone 永远 0 因为多步条件链永远凑不齐

**根因**：PPO 离散头是 16 个独立 Bernoulli sigmoid，sigmoid 全收敛到 0.5（"每个键 50% 概率"是局部最优）。**这不是参数问题，是架构问题**。盲投扩参没用。

**正确解决**（D-039）：sigmoid → softmax + multinomial，每步互斥选 1 个动作（+ NOOP）。改完 100k 步就观察到"每步累计 95% + NOOP 5%"，move 从 0.9→0.6 不再瞎跑。

**教训**：调参的第一性原则不是"加更多"，是"先抓 action histogram 看 agent 真正在做什么"。诊断便宜，盲投贵。1 小时诊断省 4 小时盲训。

## Q-022 · "改完 multinomial 还是塌缩怎么办？"（2026-05-24 16:00）

D-039 发现 multinomial 不是免费午餐。改完后发现 4 轮迭代踩 3 个坑：

| 版本 | 配置 | 100k 步看到的塌缩 |
|---|---|---|
| v1 | ent_coef=0.001（10× 下调） | last 10 wear_clothes 64.6%（按 V 不亏的免费 noop） |
| v2 | ent_coef=0.005 + 不加 cost | craft_spear 19% / feed_tame 13%（多个免费 noop 共同上涨） |
| v3 | + trigger cost 0.005/step | NOOP 98%（cost > alive 收益，agent 学到不动最赚） |
| **v4** | **ent_coef=0.005 + trigger cost 0.001/step** ✅ | **16 动作 2.55-9.78% 平均，drink 6.87%，NOOP 11%。健康** |

**关键洞察 1：multinomial 也会塌缩到"廉价 noop"**。比 Bernoulli "全 0.5" 好但不是不会塌缩。

**关键洞察 2：堵塌缩用 trigger cost 是真根因修复**。它给"按了无效"扣钱，让"按了有效"的动作（drink 在水边、craft 在有材料时）才划算。NOOP 不收费，仍是安全选项。

**关键洞察 3：cost 数值必须 100k smoke 校准**：
- alive 收益 +0.05/s = +0.00167/step
- cost 0.005 → 远 > alive 收益 → NOOP 锁死
- cost 0.001 → < alive 收益 → 鼓励行动但惩罚瞎按
- 校准方法：跑 100k smoke 看 last_n=10 的 NOOP%。NOOP 5-15% 是健康区间，>50% 太高，<3% 太低

**v4 早期成果**（200k 步）：ep 12 突破 +49.3 含 milestone +40（第一次解锁阶段），死亡率 1/22 = 4.5%（vs wander 29%）。

## Q-023 · "agent 为什么这么频繁喝水？"（2026-05-24 16:30）

**根本原因**：drink 是这游戏里**最便宜的高密度稳定奖励**。

奖励密度对比（每秒收益）：

| 行为 | 系数 | 触发条件 | 每秒收益 | 风险 |
|---|---|---|---|---|
| **drink** | +0.03/ml × 60ml/s | 站河里按 Q | **+1.8/s** | 河边狼少 |
| eat | +0.04/kcal × 偶尔 25kcal | 走到 ripe 植物按 E | +0.5~1/局 | 找植物 |
| collect | +0.5/件 | 走到资源按 R | +1.5~3/局 | 中等 |
| shelter | +5/次 | 1W+1G + 按 B | +5/局（一次性）| 低 |
| kills | +5/狼 | 用 spear 攻击 | +5/局，但被咬 -25/s | **极高** |
| milestone | +20/次 | 解锁阶段 | 一局可能 0-1 次 | — |

drink 折算成单步 +0.06/step。这是所有可控行为里最高的稳定收益。PPO 是贪心的，找到 +0.06/step 的 plateau 就锁死。

**修复尝试 D-040**：调慢 thirst 衰减 12/min → 4/min（3× 慢）。r_water 上限从 +1.8/局 → +0.6/局。

**结果**：drink 触发率从 32% → 5%（用户目标达成 ✅），但**副作用严重**：
- agent 失去"必须找水"的隐式导航信号 → 四处乱走 → 撞狼死亡率 24%（vs ppo_run3 0%）
- last 51 ep NOOP 从 22% 飙到 **52%**（一半时间什么都不做）
- craft 系列归零（craft_spear 9%→0.5%）
- r_milestone 退化到 0（之前能偶发 +40）

**教训**：单数值调整有连带效应。drink 退场后 agent 学到"NOOP + 偶尔采集"这个更糟的局部最优。下一步候选 D-041：
- 折中 thirst_decay = 6-8/min（原 12 与新 4 之间）
- 或保留 4，加避狼 reward shaping 对冲死亡率
- 或加 NOOP cost 防 agent 完全摆烂

## Q-024 · "为什么 D-043 不用真正的 PBRS？"（2026-05-24 19:30）

**真 PBRS 公式**：F(s, a, s') = γΦ(s') - Φ(s)

**Ng/Harada/Russell 1999 定理**：上面的 F 是**唯一**保证不改变最优策略的 shaping 形式。

**问题**：当 Φ(s) > 0 且保持不变（如"我有家"持续状态）时：
- F = γΦ - Φ = (γ-1)Φ = -0.01·Φ（γ=0.99）
- **每步给负 reward**！agent 会学"破坏 Φ"以避免持续惩罚

这跟用户想要的"我有家的持续感"**反着来**。

**D-043 P2 实际用的**：state-based bonus：


每步直接给 +Φ × 系数。**违反 Ng 1999**（理论上改变最优策略），但：
- 持有 Φ 越久越赚 → 学到"积累状态"
- Φ 最大 ~1.5，每步 max +0.0075，9000 step max +67
- 仍在 reward clip [-10, +10] 内

**trade-off**：放弃 policy-invariance 数学保证，换来用户要的发展感。如果观察到 agent "贪积累而不顾生存"，可降低 w。

参考： § 2 PBRS

## Q-025 · "score 是什么？比 reward 好吗？"（2026-05-24 19:30）

**score（几何均值）**：



其中 success_rate_i = "解锁第 i 个 milestone 的 episode 比例"。

**vs reward**：
- reward 衡量"agent 累积赚了多少"
- score 衡量"agent 学到多少种行为"

**几何均值的好处**：单个 milestone success rate 100% **不能**主导总分，必须广泛解锁。例：
- A: drink 100%, 其他 0% → score = (1+1)^0.1 · (1+0)^0.9 - 1 ≈ 0.072
- B: 10 个 milestone 都 50% → score = (1+0.5)^1 - 1 = 0.5

B 比 A 高 7×。这正是"故事弧"想要的——**广度优先**。

**Crafter 用 score 而不是 reward 作主指标**。D-043 起本项目也输出 score 到 jsonl 和 console。

**阈值参考**（Crafter 论文 5M step）：
- Random: 1.6%
- PPO: 4.6%
- DreamerV2: 10%
- Human Expert: 50%

我们 wander baseline ≈ 7%，2M PPO 看能到 15-25%。
