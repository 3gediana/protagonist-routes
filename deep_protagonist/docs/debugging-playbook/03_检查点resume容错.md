# 案例 03：SF 版 exe resume 旧 checkpoint 崩溃 —— 容错加载

> 一个小但典型的"改了网络结构后老存档加载不回来"的工程坑。处理方式可直接套用到任何"加了新头/新参数后要兼容旧 checkpoint"的场景。

---

## 1. 问题现象

- 加了 SF ψ 头之后，新编出的 `deep_protagonist_train.exe` 去 resume 之前的冠军 checkpoint（`runs/r1_cook_s24.pt`，pre-SF）时**崩溃**。
- 原因：新模型多了 ψ 头 → 参数集合变了 → **优化器状态（momentum/variance 等）和旧 checkpoint 对不上**，严格加载直接抛错。

## 2. 确定原因的方法

- 看崩溃栈：发生在加载优化器 state_dict 时的形状/键不匹配，不是前向/反向。
- 对照参数清单：旧 checkpoint 的 param groups 数量 < 新模型（少了 ψ 头那组）→ 一一对应的严格加载必然失败。

## 3. 修法：分层容错加载

加载策略改成"能加多少加多少，对不上就退化到安全默认"：
1. **网络权重**：能匹配的层正常加载；ψ 头这种旧档里没有的 → **fresh init**（随机初始化，靠 `sf_warmup` 训出来）。
2. **优化器状态**：如果优化器 state 和当前参数集合不一致 → **直接用全新优化器**（丢弃旧的 momentum/variance），而不是硬塞导致崩溃。
3. 其余训练计数器（step、episode）按存在即加载、不存在则归零。

这样既能 resume "已经会扑猎物"的冠军策略权重（最有价值的部分），又不会因为优化器状态对不上而崩。

## 4. 跑测试验证

- 直接用新 exe resume 旧 `r1_cook_s24.pt`：不再崩溃，训练正常起步，HUNT 行为延续（说明策略权重确实加载回来了，不是从零开始）。
- 配合 SF 自测确认 ψ 头 fresh init 后能随训收敛。

## 5. 一句话总结给接手模型

**改了网络结构（加头/加参数）后老 checkpoint 加载崩溃，几乎都是"优化器状态对不上"。修法：网络权重能加则加、新头 fresh init；优化器状态不一致就整个换新优化器，别硬加载。保住最值钱的策略权重即可，优化器 momentum 丢了无所谓（会很快重建）。**
