# 调试方法论总览（deep_protagonist）

> 这份文档不是讲某一个 bug，而是讲**遇到任何指标卡死时，应该怎么想、怎么查、怎么验证**。
> 写给接手的模型：**先读这篇，再读具体问题篇**。本目录每个 `0X_*.md` 都是按同一套方法走完的真实案例。

---

## 0. 一句话原则

**永远不要靠"猜 + 改参数 + 重跑"来推进。先埋点、先量、先用最便宜的实验把"根因"钉死，再动手改。每一次改动都必须能被一个事先想好的指标证伪。**

这条原则来自一个血的教训：DP 的 `kills=0` 卡了很久，历任都在调奖励/熵/课程/稀缺度，因为大家**默认**问题是"探索不够"或"奖励不对"。真相是：智能体根本靠近不了猎物（接近距离地板 9m，攻击需要 2.5m）。**所有调奖励的努力都作用在"靠近之后"的环节，而瓶颈在"靠近"本身。** 没人去量"它到底离猎物多近"，所以谁都没发现。

## 1. 标准排查流程（六步）

### 第 1 步：把"症状"写成一句精确的、可证伪的话
- ❌ 模糊："打猎学不会。"
- ✅ 精确："训练了 228 个回合，`kills` 这个奖励通道**恒等于 0**，`score` 卡在 0.22 不动。"

精确陈述会逼你问："我凭什么知道 kills=0？我有没有在量它？" 很多时候你会发现**你根本没在量关键量**。

### 第 2 步：先怀疑"测量"，再怀疑"系统"
卡死的指标，先问三件事：
1. 这个指标是**在哪里、什么时候**写盘的？（DP 的 PE 项目就吃过亏：振幅只在最后 20 代写，长跑前期根本读不到——见 PE 篇 `03_度量盲点.md`）
2. 我读的指标是不是**有偏**的？（早期我只在"攻击瞬间"量离猎物的距离，那是有偏的——智能体只在自认为能打到时才按攻击键。改成**每一步都量**真实最近距离后，才看到 9m 的地板。）
3. 有没有"混合桶"掩盖了真实信号？（reward.total() 里 `vital` 通道把 decay+danger+action_cost 三样混在一起，单看总分看不出哪样在拖后腿——SF 篇 `02` 里把它拆成独立线性通道才能 re-weight。）

### 第 3 步：用"全行为面板"或"预判器"在分钟级看到信号
不要跑满 48M 步再看结果。DP 的杀手锏是 **23 动作优势面板**：每隔几千步打印每个离散动作的采样次数 n + 平均优势 advantage。几分钟就能读出：
- 哪些动作被频繁采样、哪些 n=0（被 gate 锁死从没触发）；
- 被采样的动作优势是正还是负（策略想不想做它）。

这个面板一上来就推翻了"探索不够"的假设：HUNT 动作采样占比一度 47.6%——**智能体疯狂在打猎**，根本不是不探索。问题在别处。

### 第 4 步：加**针对性埋点**，把断点逐层切出来
当面板说"它在拼命做 X 但 X 没产生结果"时，加埋点把 X 这条链拆成几段，每段一个可观测量。打猎链路被我切成三段，每段一个埋点（见 `01` 篇）：
- `hunt_hits`：长矛真正命中猎物的次数 → 切"瞄准/命中"。
- `closest_prey`（每步量，不是攻击瞬间）→ 切"接近"。
- `closest_rest`：智能体离**正在力竭休息**的猎物的最近距离 → 切"有没有可乘之机/有没有利用机会"。

埋点的设计要让结果**互斥可判**：
- hits>0 & kills=0 → 卡在补刀（第二下）。
- hits=0 & closest≤2.5m → 卡在瞄准（攻击锥没对准）。
- closest>2.5m → 卡在接近（根本没贴上去）。

### 第 5 步：读数据，让数据指认根因——并且**用"改了没动"当证据**
"改一个旋钮，指标没动"**本身就是高价值信息**。打猎这案子里，我连续从三个独立方向改：
- 感知（把猎物固定进观测槽位 D-124）→ 9m 地板没动；
- 动机（趋近奖励 shaping 0.1→0.5，5 倍）→ 中位数降了但 9m 地板没动；
- 机制（耐力狩猎）→ 9m 地板还是没动。

**三个正交方向都撬不动同一个 9m 地板** → 强烈指向"这是一个内建的机械平衡点，不是学习/感知/动机问题"。这个推断只有在你**系统地、一次只动一个变量**地做实验后才成立。这就是把"改了没动"用成证据的方法。

### 第 6 步：定位根因后，在"地基红线"内找最小改动
地基锁死（**网络 I/O 不动、动作空间 23 不变、观测 582 维不变**）。所以：
- 想给智能体"猎物去向"信息 → 不能加维度 → 改观测**槽位填充优先级**（把第 0 槽固定给最近的鹿/兔，照搬狼槽位 D-048 的先例），0 维度变化。
- 想让耐力狩猎真正可用 → 不改速度（用户红线：动物必须比人快）→ 改"力竭"的触发逻辑：从"每段冲刺计时、一脱离触发距离就重置"改成"累积疲劳、脱离后只缓慢恢复"（D-126）。

## 2. 反 Goodhart 纪律

不要为了让某个指标变好而发"赏金"。例如：**绝不**因为 kills=0 就直接对每次击杀发大额奖励——那会让策略学会"刷击杀数"而不是学会真实的狩猎能力，并污染其它行为。正确做法是**修复让狩猎在物理/认知上可达的底层机制**，让击杀作为"坚持追 + 瞄准"的自然结果挣出来。势能塑形（PBRS, `phi_hunt_progress`）是允许的，因为它不改变最优策略、只加速收敛。

| 23 动作优势面板 | 分钟级看每个动作的采样 + 优势 | 训练日志里直接打印，看 `act[i] n=.. adv=..` |

## 4. 给接手模型的最后叮嘱

1. **看到指标卡死，第一反应是"我有没有在量正确的中间量"，而不是"我该调哪个参数"。**
2. **每次只动一个变量**，并预先写下"如果它是根因，这个埋点应该变成什么样"。
3. **改了没动 = 排除了一个假设**，记下来，别重复试。
4. **守地基**：582 维 / 23 动作 / 网络 I/O 是红线，所有解法在红线内找（改填充优先级、改机制逻辑，而不是加维度/加动作）。
5. **频繁存档 + 拉回**：训练 checkpoint 要定期 rsync 回 Windows，临时机随时可能没。
6. 具体案例见同目录 `01`–`03`。
