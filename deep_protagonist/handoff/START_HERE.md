## 2. 这是什么类型的活：操作环境（接手模型最容易卡的地方）

### D-134 两阶段狩猎机制（用户设计，是本轮最值钱的一条，务必理解）
之前狩猎对**任何**猎手都几乎打不赢（连完美信息脚本老师 202 局 0 杀），因为：①开阔地猎物比人快、12m 就炸毛窜走回血，追不到力竭；②贴脸也扎不中（朝向锥 + 多猎物分摊伤害）。用户的解法把它变成清晰状态机：
- **猎物两阶段**：平时只走路（≤主角速度）、主角靠近**不再恐慌逃跑**→能正常走到身边下第一矛；**被矛扎中后**才冲刺爆发（很短 ~0.35s≈2m）→然后原地停下喘 → 留出补第二矛的窗口。两矛(50+50)毙命。
- **主角两阶段**：移动是连续向量，推杆力度 **≤0.6 走路 / >0.6 冲刺**（最高 6m/s，超 4 掉体力）。何时冲由策略自己决定。**没加新动作维度** → 基线 checkpoint 还能 resume（关键：不用从零重训）。
- 全挂在 `DP_HUNT_V2` 开关后（默认关 = 游戏分毫不变）。

## 4. 当前最重要的诊断：P0 涌现谱（数据基础，别重测）
完整数据见 `docs/handoff/P0_baseline.md`（或项目根 `P0_涌现谱_baseline.md`）。一句话结论：

> **两个系统都坍缩到约 1/3 的机制宽度，撞的是同一面墙。**

- **DP**：23 个动作只有 **8-10 个**真被用到；10 个里程碑平均只解 **2.3-2.9** 个。**整条科技树(cook/mine/axe/pickaxe/monument)、造屋、种地、驯养全是 0%**。存活率 0（绝大多数**冻死**）。
- **"空按"现象（关键）**：`ft_bc10` 每局狂按 `tendfire` **416 次却照样冻死** = 按了火但没真取暖。所以有效宽度比 8/10 还窄。
- **PE**：4 个核心机制只有"投掷"真能成(19.6%)；砍树/造物/近身狩猎各试 5-6 万次，成功≈0。

**根因（不是模型笨）**：绝大多数机制要么**机制上够不着**（成功率≈0），要么**够得着但没收益压力非做不可**（不喝水/不生火/不造屋暂时也死不了）→ 各自坍缩成一条最省力活路。这就是"**标量爬坡 → 单策略坍缩**"。

## 8. D-136 当前实验：打破 2.7/10 广度天花板（接手时正在跑）
P0 之后用户拍板"全权自主决策、全面抓多机制"。本轮做的：

1. **实现了广度牵引 `phi_repertoire`**（`--ppo-repertoire-shaping`）+ 高熵探索 → 把一局用到的机制数从狩猎专精的 2.3 恢复到 **~2.7/10**，然后**到顶**。涨上来的全是"本来够得着"的（进食/喝水），真正想要的遮蔽棚/播种/收成/随从/房子仍≈0%。
2. **钉死了 2.7 天花板的总根因 = 冻死**：agent 每局 ~6000 步就冻死，没活够久去把多步机制串完。而**除穿衣外所有取暖方式（生火/棚子/房子）都要站着不动**，跟它"边走边采"的主流活法冲突 → 点了火就走开、然后冻死。**只有穿衣是"可携带的暖"**。
3. **试过重启遮蔽棚（`DP_LEANTO`）→ 实测无效**：那个 `place_shelter` 动作(idx4)被基座策略 `ft_bc10` 学成了死动作（D-089 曾把它设成空操作），120 局**一次都没被选中**，logit 压死、高熵也救不活。**教训：对一个策略从不选的动作做机制改动 = 白做。先量动作选择率再改。**
4. **当前在跑：保暖牵引 `--ppo-warmth-shaping`**（势函数=体温恢复）。理论：想躲掉降温惩罚 → 策略被拉去穿衣（唯一跟边走边采兼容的暖）。两个变体 `ft_wa`(coef 2) / `ft_wb`(coef 4)，都从 `ft_bc10` resume，配置 = rep 1.5 + hunt 0.3 + ent 0.02 + warmth。**盯的信号：mean_unlocks 顶过 2.7 + clothing 触发率从 ~5% 往上 + avg_steps 从 ~5900 往上（冻死减少）**。用 `bmon.ps1 -Tags wa`（每 tag 单独跑，数组绑定有坑）。

### PE 的平行发现（D-136，重要）
PE 撞的近身狩猎墙**跟 DP 不是同一种**，DP 的两阶段狩猎修复**不能直接移植**：PE 的"猎物"（食草/采集生态位）**本身就是 NEAT 进化出来的智能体**，用自己的神经策略逃跑（会随主角共同进化越逃越远），不是脚本动物，没有一个"逃跑速度"常量可改。PE 的 `chop`（砍树）低成功率则跟 DP 旧的"伤害摊到一整群"同型——见 `ChopTreeCapability.cpp`，换树会重置 session（但树的血量在 TreeLayer 里持久，所以根因更可能是"贴树时间不够/树太多太分散"）。**PE 的机制杠杆要改 fitness 权重(.toml)或 capability 代码 + 重编重启**，比 DP 的在线实验风险高，动前务必先量。

### 突破：unlocks 2.8 → 3.33/10（打破了卡死的天花板）

**根因诊断（已钉死）**：会打猎的底座 `ft_bc10` 的 23 个动作里只有 8 个是活的，
`shelter/plant/cook/mine/...` 这些动作的概率被压成 0（死动作）——奖励再大也点不亮一个死动作。
这就是过去卡在 2.8 的硬上限。光给奖励 + 高熵探索（保暖势函数 wa/wb/wc 三变体）干净地证明了这一点：三个变体全部卡在 2.8/10、活动作恒为 8/23。

**解法（已验证有效）**：用项目自带的 BC→PPO 管线，**合并两个冠军的示范**做出一个"全能底座"：
- 狩猎专精 `demos_hv2_s24.bin`（90 万条，hunt/collect/deposit/tendfire 活）
- 烹饪/保暖专精 `demos_cook_s24.bin`（62.5 万条，drink/sleep/grassdress/blueprint/water/feedtame 活）
- 合并 152 万条 → BC 出 `runs/bc_gen.pt`（48.8MB，3 epoch，disc_ce 5.01）。
- 冻结评测确认：**复活了 5 个原本死掉的动作**（drink/sleep/grassdress/blueprint/water），活动作 8→10+。

**PPO 微调结果 `runs/ft_gen.jsonl`**（从 bc_gen 起，breadth 1.5 + warmth 2.0 + ent 0.02）：
- **unlocks 稳定 3.33/10**（基线 2.8）；6 种机制 98–100% 触发：喝水/进食/造矛/穿衣三件套(草衣+毛皮+穿戴)/睡觉/浇水。
- 整局全存活 0 → 4/692 episodes（冻死减少）。

**接手直接用**：最佳策略 `runs/ft_gen.pt`；全能底座 `runs/bc_gen.pt`。
