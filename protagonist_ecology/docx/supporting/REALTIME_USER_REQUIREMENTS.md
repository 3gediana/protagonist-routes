# REALTIME_USER_REQUIREMENTS.md · 用户实时需求汇总

> 说明：这份文件用于持续汇总用户在当前会话里提出的实时想法与目标效果，供另一个正在开发项目的 AI 直接读取和执行。
>
> 规则：
> - 这里的条目默认都按 **必须实现的硬需求** 处理，不要擅自降级成“可选建议”。
> - 用户原话往往是口语化、压缩的；这里已经做了工程化扩写。
> - 除非用户后续明确推翻或修改，否则旧条目继续有效。
> - 实现时应优先保证“真实机制存在并驱动行为”，不要只做表面数值或装饰性 UI。

## R-001 · 全体个体统一生理状态系统

### 用户原意

用户希望当前 route 中的“血量、口渴、核心体温、饥饿”等状态，像游戏角色那样是真实存在于系统中的个体级数值，而不是只有少量抽象代理指标。用户明确要求：

- 不只是主角，**所有活着的个体** 都必须有这套生理状态。
- 个体应根据自身状态值变化做出行为，而不是恒定模式行动。
- 低于不同阶段阈值时，应触发不同强度的 debuff，而不是只有单一硬阈值。
- 季节必须能真实影响这些核心生理值。
- 长时间运动应显著增加口渴与饥饿消耗；如果当前没有饥饿系统，则必须补上。

### 当前代码现实（供开发 AI 校准，不要误判为已完成）

截至当前代码现实：

- 已存在的 route 级真实状态主要只有 `health` 与 `hydration`。
- 当前 `SurvivalStatusLayer` 只跟踪 protagonist，不是所有活体个体。
- 当前还没有统一的 `hunger` 独立数值。
- 当前还没有统一的 `core temperature` / `body temperature` 独立数值。
- 当前 `season` 层已存在，但尚未与口渴 / 饥饿 / 核心体温衰减建立真实耦合。
- 当前运动主要消耗 `energy`，还没有把运动时长 / 强度显著耦合到口渴、饥饿、体温变化。
- 当前脱水 / 受伤更接近单阈值逻辑，还不是多阶段生理 debuff 系统。

### 必须实现的目标

需要把当前的零散生存机制升级成一套 **全体个体统一生理系统**，至少包含：

- `health`
- `hydration`
- `hunger`
- `core_temperature`

其中：

- **health**：最终生命安全值，不应再承担所有生理逻辑本体。
- **hydration**：真实口渴/含水状态，影响行动能力与健康风险。
- **hunger**：真实饥饿/饱腹状态，必须独立存在，不能仅拿 `energy` 敷衍替代。
- **core_temperature**：核心体温，必须是真实连续数值，不是简单“冷/不冷”布尔量。

### 作用对象范围

这套系统必须作用于：

- protagonist
- 所有 background species
- 后续新增的活体个体

也就是说，**不能再出现“只有主角有真实生理系统，背景物种只是简化 energy 盒子”** 的情况。

### 行为驱动要求

个体行为必须由自身生理状态真实驱动，而不是只由外部奖励或脚本式优先级驱动。至少应满足：

- 口渴上升后，个体更倾向找水、回水源、缩短高消耗活动。
- 饥饿上升后，个体更倾向找食物、改变觅食/捕猎/搬运优先级。
- 核心体温过低时，个体更倾向靠近火源、巢穴、避风处，减少暴露。
- 核心体温过高时，个体更倾向降温、远离高热环境、降低长时间高强度运动。
- 这些行为变化应是连续增强的，不应只有“低于某个点突然像脚本一样切模式”。

### Debuff 机制要求

必须把当前单阈值式生存惩罚升级为 **多阶段阈值 + 连续恶化** 机制。

示例要求：

- `hydration` 低到轻度区间：开始出现轻微负面影响。
- 低到中度区间：移动/判断/恢复效率进一步下降。
- 低到重度区间：显著受损，并开始快速转化为健康风险。
- `hunger`、`core_temperature` 也应有类似多阶段衰减结构。

重点要求：

- debuff 应该是 **分阶段增强** 的。
- 各生理值的严重恶化应可相互叠加。
- 最终 `health` 更像这些长期失衡的综合后果，而不是所有系统直接绕过中间层只改血量。

### 季节与环境耦合要求

季节必须真实改变核心生理系统，而不是只做视觉或上下文标签。

至少要实现：

- **冬季**：核心体温下降更快，保温需求明显增强。
- **夏季**：核心体温上升更快，散热需求明显增强。
- 不同季节下，水分与食物压力也可以不同，但至少体温这一层必须真实耦合。

进一步要求：

- 如果同时存在昼夜、地形、洞穴、营火、巢穴、水域等环境机制，应允许这些因素共同影响个体体温变化。
- 也就是说，`season` 不应独立悬空存在，而应进入真实生理更新方程。

### 运动负荷耦合要求

长时间运动或高强度运动必须带来更高的生理消耗，至少包括：

- 运动越久，`hydration` 下降越多。
- 运动越久，`hunger` 下降越多。
- 高强度或持续运动会改变 `core_temperature`。

用户明确想要的不是“站着和狂奔几乎一样耗状态”，而是更接近真实生命体的负荷变化。

### 感知与决策要求

每个个体必须能够感知自己的这些生理状态，且这些状态应进入其决策回路。

至少需要保证：

- 自身生理状态进入感知/输入，而不是只存在于世界层但脑拿不到。
- 背景物种也不能被排除在外。
- 后续指标和 trace 应能证明这些状态真的参与了行为变化。

### 指标、Trace 与验证要求

实现后必须补足可验证证据，避免“机制写了，但外部看不出来”。至少需要：

- 每个个体级 trace 中可看到 `health / hydration / hunger / core_temperature`。
- 汇总指标中能看到全体活体个体的这些状态统计，而不是只看 protagonist。
- 能区分轻度 / 中度 / 重度生理压力人数或比例。
- 能验证季节切换前后，核心体温变化速率确实不同。
- 能验证高运动负荷个体比低运动负荷个体更快消耗口渴/饥饿，并出现相应行为变化。

### 验收标准

另一个开发 AI 实现这条需求时，至少应达到以下验收结果：

1. 所有活体个体都拥有统一生理状态，而不是 protagonist-only。
2. `hunger` 与 `core_temperature` 作为真实独立数值存在。
3. 生理状态具有多阶段阈值与逐步加重 debuff，而不是单点触发。
4. 冬夏季节会真实改变核心体温演化速度。
5. 持续运动会显著增加口渴与饥饿消耗，并影响体温。
6. 个体会因这些内部状态变化而表现出对应求水、觅食、取暖、避热等行为倾向。
7. trace / metrics 能清楚证明上述机制已经真实接入，而不只是代码里“名义存在”。

### 实现优先级

高优先级。

这是底层生命机制，不是装饰性 feature。若这层不真实，后续很多“像活物一样”的行为都会显得假。

## R-002 · 生产世界必须是可参与、可干预、持续演化的活生态系统

### 用户原意

用户希望所谓“生产世界”不是一个只能旁观的静态展示场，而是一个**用户能够实际进入、实际施加影响、并持续观察系统如何响应与演化**的长期运行世界。用户明确要求：

- 用户应能向系统“投放”事件，直接影响世界。
- 系统里的生物应当对这些刺激作出响应，而且是所有生物，不只是主角。
- 在生产系统中，生物也应能够继续进化自己的网络，而不是上线后完全冻结。
- 最终理想画面是：用户可以像“玩家”一样进入世界，近距离观察这些生物。
- 这些生物需要有较精致的 3D 模型和动作表现。
- 四季更替时环境应真实变化。
- 地形差异化应明显。
- 各种行为之间应存在关联，不能像彼此孤立的功能开关。

### 核心定位要求

生产世界的定位必须明确为：

**一个用户可参与干预的长期生态运行环境**，而不是“训练结束后的只读回放场景”。

这意味着生产世界至少同时承担以下角色：

- **持续存在的生态环境**：世界不是短时 demo，而应支持长期运行。
- **用户参与界面**：用户不是只能看，而是能向世界施加影响。
- **行为观察场**：用户能近距离观察个体、群体、季节、地形、互动的真实变化。
- **持续学习 / 持续演化场**：系统上线后仍具备继续适应和继续变化的能力。

### 用户参与与事件投放要求

必须为生产世界设计一套明确的“用户干预入口”，允许用户像玩家或世界干预者一样向系统投放事件。

这些事件不应只是装饰性特效，而应是真正进入世界机制并改变生态状态的输入。至少应支持以下方向：

- 投放环境事件
- 投放资源变化
- 投放危险或扰动
- 改变局部地形或通行条件
- 改变局部气候 / 温度 / 季节压力
- 向局部区域引入新的机会或威胁

关键要求：

- 这些投放事件必须被世界中的真实机制消费。
- 事件必须能改变个体处境，而不是只在 UI 层显示一下。
- 用户干预后，应能从行为与指标中看到系统响应。

### 全体生物响应要求

用户明确要求：**系统里面的生物（所有）理应对这些刺激做出响应**。

因此必须避免以下错误实现：

- 只有 protagonist 对事件有反应，背景物种无感。
- 只有脚本单位会响应，进化个体不响应。
- 事件只影响数值统计，不影响个体决策。

正确要求应是：

- 所有活体个体都能感知与自己相关的外部刺激。
- 不同物种、不同状态、不同位置的个体可出现不同响应。
- 事件影响应沿着“感知 -> 内部状态 -> 决策 -> 行为 -> 生态后果”完整传递。
- 响应应当允许短期应激和长期适应两层同时存在。

### 生产环境持续进化要求

用户明确希望：在生产系统当中，生物也要能一直进化自己的网络。

因此生产世界不能被定义成“训练结束后参数永远冻结”的纯展示模式。必须支持某种形式的**持续进化 / 持续学习 / 持续更新**机制。

这里的重点不是限定只能采用某一种技术路线，而是必须满足结果要求：

- 生物上线后依然能够继续适应环境与用户干预。
- 网络结构、参数或族群分布应允许继续变化。
- 世界中的长期变化不能永远只靠训练前学到的旧知识硬撑。

但实现时必须同时满足稳定性要求：

- 不能让生产世界因为无门槛在线自我修改而迅速崩坏。
- 需要保留版本管理、回滚、影子评估或分层放行等安全机制空间。
- “持续进化”必须是可观测、可审计、可约束的，而不是黑箱漂移。

### 玩家进入世界的目标体验

最终目标体验必须包含“用户像玩家一样进入这个世界”。

这不是单纯开一个 free camera 就算完成，而是要求形成**玩家尺度的近距离观察体验**。至少应支持：

- 用户进入世界中的一个具体观察视角或角色视角。
- 用户可以近距离看个体如何移动、觅食、避险、协作、受环境影响。
- 用户能够感受到个体不是抽象点，而是具有存在感的生物。
- 用户干预世界后，可以在局部现场直接观察连锁反应。

### 3D 模型与动作表现要求

用户明确要求最终看到的是较精致的 3D 生物，而不是长期停留在简陋占位体。

因此最终展示层必须朝以下方向实现：

- 生物具有相对精致的 3D 模型。
- 不同类型个体在体型、轮廓、动作气质上有区分。
- 动作系统不能只有滑动位移，至少要有基本可信的运动、转向、待机、觅食、攻击/逃避、交互等表现。
- 行为动画应尽量与内部状态或实际行为对应，避免“模型在演，机制在另一套世界里跑”的割裂感。

### 四季与环境视觉变化要求

用户希望四季更替时环境本身也会变化，因此四季不能只体现在后台参数表上。

必须追求两层同时成立：

- **机制层变化**：季节真实影响环境压力、资源条件、生理负担等。
- **视觉层变化**：用户能直接看出季节切换带来的环境差异。

可接受的目标方向包括但不限于：

- 植被/地表色调变化
- 水体、积雪、干燥度、能见度或天气感变化
- 营火、巢穴、遮蔽物等环境设施在不同季节的重要性变化

### 地形差异化要求

用户明确要求地形差异化要明显，因此地形不能只是远看有起伏、近看行为无差别的背景板。

必须让地形同时在以下层面显著存在：

- **视觉存在感**：不同区域一眼能看出明显差异。
- **通行差异**：移动成本、路径选择、可达性不同。
- **生态差异**：资源、藏身、暴露、危险分布受地形影响。
- **行为差异**：个体在不同地形上的决策与生存方式不同。

### 行为关联性要求

用户明确要求“各种行为之间要有关联”。这意味着系统不能做成一堆互不相干的小 feature。

必须让行为之间形成因果链和联动关系，例如：

- 季节变化影响体温与资源，再影响迁移、觅食、取暖与合作。
- 地形变化影响寻路与暴露风险，再影响觅食、逃跑、捕猎与建造。
- 用户投放事件改变局部压力，再影响群体聚散、通信、竞争与恢复。
- 持续进化进一步改变个体如何解释这些压力与机会。

最终要求是：用户近距离观察时，能够感到“这个生态系统里的事情是互相牵动的”，而不是每个模块各演各的。

### 验收标准

另一个开发 AI 实现这条需求时，至少应达到以下结果：

1. 生产世界被明确实现为可长期运行、可用户参与干预的系统，而非只读展示。
2. 用户能够向世界投放会真实改变机制状态的事件。
3. 所有生物都会对这些刺激产生可观察响应，而不是 protagonist-only。
4. 生产世界中存在持续进化 / 持续学习 / 持续更新能力，而不是彻底冻结。
5. 用户能够以玩家尺度近距离观察世界中的个体与群体。
6. 最终展示目标明确包含较精致的 3D 模型与动作，而非长期停留在占位表现。
7. 四季更替能同时体现在机制层与视觉层。
8. 地形差异在视觉、通行、生态、行为上都足够明显。
9. 不同行为机制之间能形成可观察的联动与因果链，而不是功能孤岛。

### 实现优先级

高优先级，但属于**路线级终局定位约束**。

这条需求不一定要求马上把完整 3D 玩家体验一次性做完，但后续所有 live / production / world / interaction / visualization 设计都必须以这个方向为准，不能把生产世界收缩成“只做个观察窗口”或“只做个静态演示厅”。

## R-003 · 机制迭代必须采用“分层验证 + 兼容迁移 + 小任务重训”工作流，不能依赖旧训练模型始终可用

### 研究结论

- **NEAT**：从最小结构开始，靠 `speciation` 保护结构创新，再逐步复杂化。
- **Incremental Shaping / Scaffolding**：难任务先在简化世界或低保真环境证明可学，再升到复杂世界。
- **Transfer / Reuse**：旧控制器可参考，但只在任务语义接近时有效；不能默认所有机制改动都能平滑继承。
- **Progressive / Modular 思路**：高风险升级优先保留旧路径，再加新分支，而不是一次性覆盖。
- **反例提醒**：增量进化不是永远更好，必须用机制级实验验证，不能靠信仰。

### 项目硬规则

1. 不以“旧训练大模型还能不能用”作为新机制立项前提。
2. 每个重要新机制都必须先有**无训练验证**，再有 **micro-task 短训**。
3. 能兼容就兼容扩展；不兼容就明确 `version gate`，允许 `fresh start`。
4. 只有已在小任务中证明有价值的机制，才进入主线长训。

### 变更分级

- **A · 接线改动**：trace、metrics、事件、日志；不要求重训。
- **B · 兼容扩展**：加层、加分支、加状态，但旧 action 语义不变；允许兼容 resume。
- **C · 半不兼容**：输入布局或 action 语义明显变化；必要时做行为回放或 teacher-student 迁移。
- **D · 不兼容**：genotype、brain schema、核心状态语义重写；直接 `fresh start`。

### 验证层级

- **L0 单测**：parser、layer、event、perception、ledger。
- **L1 脚本验证**：固定动作 agent、规则 agent、runtime 注入、traceprobe。
- **L2 micro-task**：小地图、短 episode、少变量，强迫该机制发力。
- **L3 集成 smoke**：并入 route 短训，看是否打崩现有系统。
- **L4 正式长训**：仅在前 4 层通过后再做。

### 当前项目的直接落地

- **通信语义**：先做 `scout-worker`、`danger-warning`、`build-request`。
- **统一生理系统**：先做 `thirst-return`、`thermoregulation`。
- **因果信用分配**：先做 `signal -> ally action -> reward attribution` 的 traceprobe。
- **连续时间**：先单独验证 `move/eat/drink/chop/build` 的持续时间语义。
- **goal / world model**：先在固定场景验证 persistence / predictor 是否提高稳定性。

### 需要补的基础设施

- `micro-task` configs
- `traceprobe` configs
- mechanism smoke configs
- mechanism ablation harness
- checkpoint / schema versioning
- behavior replay set

### 验收标准

1. 新机制第一轮验证不再依赖旧大模型。
2. 每个关键机制都有 L0-L2 路径。
3. 集成前先声明兼容级别。
4. D 类改动允许旧 checkpoint 作废。
5. 长训预算只投给已通过小任务验证的机制。

### 实现优先级

高优先级。

这是后续所有机制研发的默认工作流。

## R-004 · 新机制必须先定义完整证据链，再开发、统计、分析，禁止单指标断言生效

### 核心要求

后续任何新机制在进入正式开发前，必须先写出一份**机制证据链 spec**。没有这份 spec，不得把该机制视为“已验证”或“已生效”。

### 禁止事项

- 禁止只凭单个结果参数就断言某机制生效。
- 禁止只有终局指标，没有中间链路证据。
- 禁止出现“结果变好了，但不知道机制到底有没有被真正使用”的实现。

### 开发前必须定义的完整链路

每个机制都必须明确以下 7 段：

1. **触发条件**：机制在什么情况下应被激活。
2. **输入消费**：它读取哪些感知、记忆、事件或状态。
3. **内部变化**：它改变了什么内部变量、记忆、belief、goal 或 ledger。
4. **决策偏转**：它应如何改变 action 选择或目标切换。
5. **外显行为**：屏幕上应能看到什么行为差异。
6. **环境后果**：行为应给世界或其他个体带来什么变化。
7. **结果收益**：最终在哪些指标上体现收益或代价。

### 每一段都必须有证据字段

对上面 7 段，开发 AI 必须在实现前就声明：

- 对应的 `trace` 字段
- 对应的 `metrics` 字段
- 对应的事件日志或回放证据
- 预期变化方向

如果链路中的某一段没有可观测字段，就必须先补观测，再谈机制验证。

### 标准证据结构

每个机制最终至少要交出以下 6 类证据：

- **使用证据**：机制是否真的被触发和读取。
- **状态证据**：内部状态是否真的变化。
- **决策证据**：policy / goal 是否真的偏转。
- **行为证据**：外显行为是否真的改变。
- **结果证据**：终局指标是否改变。
- **消融证据**：关闭该机制后，这条链是否明显断裂。

### 判定规则

- 只有“结果证据”没有中间链路时，结论只能写 **possible**，不能写 **confirmed**。
- 至少同时具备“使用 + 决策/行为 + 结果 + 消融”四类证据时，才允许写“机制生效”。
- 如果结果变好，但链路中段缺失，默认按**证据不足**处理，不得宣传为成功实现。

### 对开发流程的强制要求

以后每个新机制都必须按以下顺序推进：

1. 先写机制证据链 spec。
2. 再补足 trace / metrics / event instrumentation。
3. 再做 L0/L1 验证，确保链路能被观测。
4. 再做 micro-task / traceprobe / smoke。
5. 最后才做结果分析和是否进入长训的决策。

### 当前项目的直接落地方式

- **通信语义**：`signal emit -> signal receive -> listener goal/belief change -> route/action change -> 协作到达/撤退 -> 成功率变化`
- **统一生理系统**：`生理压力上升 -> goal shift -> 找水/取暖/避热 -> 状态恢复 -> 生存/效率变化`
- **因果信用分配**：`事件发生 -> ledger 记账 -> 奖励差异 -> 后续策略复用变化`
- **world model**：`预测输出 -> goal/path change -> 风险回避或资源回收改善 -> 成功率变化`

### 交付物要求

每个机制至少要有：

- 一张机制证据链表
- 一组对应的 trace / metrics 字段
- 一条 micro-task 或 traceprobe 验证路径
- 一份消融对照
- 一份明确写出 `confirmed / possible / unsupported` 的结论

### 验收标准

1. 新机制开发前先有完整逻辑链路 spec。
2. 新机制分析不再只看单个总表参数。
3. 关键链路节点都有观测字段。
4. 每个机制都能给出从触发到结果的连续证据。
5. 没有消融和链路证据时，不得写“机制已生效”。

### 实现优先级

立即执行。

这是后续所有新机制开发的强制分析规范。

## R-005 · 每个物种都必须有鲜明、稳定、肉眼可辨的生态特点，不能只是参数变体

### 用户要求

用户希望当前项目里的“物种”不是只有速度、攻击力、能量之类的数值差别，而是**真的表现出符合直觉的生态特点和社会规则**。用户明确希望看到类似：

- 一山难容二虎：强势独居顶级掠食者会占地、排斥同类、避免共享猎场。
- 狼成群结队：群猎型物种会聚群、跟随、包夹、协同围猎。
- 秃鹫飞天伺机而动吃尸体：食腐型物种会盘旋、观察、等待尸体，再下降取食，而不是像普通掠食者一样主动硬追猎物。
- 人类团结围猎：主角/人类系必须体现协作、信号、分工、围猎、工具与集体收益。

### 核心要求

每个物种都必须具备自己的**生态人格**，至少在以下层面形成稳定差异：

- 食性与目标偏好
- 同类关系规则
- 风险偏好
- 领地/聚群/游走方式
- 战斗或逃跑风格
- 资源利用方式
- 对尸体、危险、信号、同伴的响应方式

这些差异必须长期稳定存在，用户观察几分钟后就能明显感到“这不是同一种东西换个皮”。

### 必须避免的错误实现

- 只是移动速度、血量、攻击力不同，行为气质却几乎一样。
- 物种名字不同，但决策逻辑几乎完全相同。
- 所谓“群居”“独居”“食腐”“围猎”只写在文档里，屏幕上看不出来。

### 结果要求

用户最终应该能凭观察直接分辨：

- 哪些物种是独居强势掠食者
- 哪些物种是成群活动的围猎者
- 哪些物种是高空徘徊、机会主义食腐者
- 哪些物种是会合作、会组织、会围猎的人类系

### 实现方向参考

- 独居顶级掠食者：参考领地型虎类 ABM，关键不是单纯高攻击，而是固定活动圈、同类排斥、领地重叠冲突。
- 群猎者：参考狼群简单规则围猎研究，先做“逼近猎物 + 接近后与同伴拉开 + 短时共享目标”，不必先上复杂通信。
- 食腐者：参考秃鹫 producer-scrounger 研究，做高空巡航、观察他者下降/聚集、尸体优先取食，不给全图尸体雷达。
- 人类系：参考协作围猎研究，把环境做成单人难杀、多人围堵收益高、信号/工具/分工明显提高成功率。
- 总原则：物种特点优先靠世界规则、局部感知、收益结构和生态位差异逼出来，不靠名字和数值换皮。

### 实现优先级

高优先级。

这属于世界真实感和生态观感的核心要求。

## R-006 · 所有活体都必须进入生老病死与交配繁殖闭环，且繁殖要真实影响基因与种群

### 用户要求

用户希望这个世界里的活体不是一批静态模板单位，而是**真的会出生、成长、衰老、生病、死亡，并且能够交配繁殖**。

繁殖不能只是简单复制个体，而应体现：

- 父母基因融合
- 突变
- 后代差异
- 种群长期变化

### 核心要求

所有活体都必须具备至少以下生命周期阶段：

- 幼体
- 成体
- 老年
- 死亡

并且以下机制必须真实进入系统：

- 年龄增长
- 年龄对移动、战斗、繁殖、生存能力的影响
- 疾病、受伤、体弱、恢复
- 自然死亡与非自然死亡
- 可触发繁殖的配对条件
- 繁殖后的后代生成

### 繁殖要求

繁殖必须满足以下结果：

- 不是无成本刷怪式增殖
- 至少受年龄、状态、距离、物种兼容性或环境条件约束
- 后代基因来自父母融合，而不是单纯复制一方
- 后代允许产生突变
- 后代会真实进入世界、参与后续竞争、生存与演化

### 结果要求

用户最终应该能看到：

- 物种不是永远固定数量和固定面貌
- 不同世代会有差异
- 老病死会改变群体结构
- 繁殖会真实改变种群分布与行为风格

### 必须避免的错误实现

- 只有死亡，没有成长和衰老
- 只有生成新单位，没有真正交配与基因融合
- 突变只写在参数里，但种群长期看不出变化
- 生命周期只影响后台数值，不影响可观察行为

### 实现方向参考

- 生命周期：参考 age-specific selection / ageing 研究，做按年龄变化的生存率、繁殖率和行为能力，不要只做“老了统一扣血”。
- 疾病：参考 wildlife individual-based disease 模型，做近接触传播、尸体/污染传播，并让年龄、营养、拥挤度影响易感性。
- 繁殖：参考 NEAT 与 Avida 的性繁殖/重组思路，做择偶条件、双亲基因融合、突变、后代入世，而不是单亲复制。
- 后代：幼体必须比成体弱，需要成长，不能出生即满战力。
- 代际：必须保留 parent ids、generation、lineage，让种群变化和行为分化可追踪。
- 总原则：繁殖不是刷怪，死亡不是直接消失，尸体、疾病、后代都要进入同一生态闭环。

### 实现优先级

高优先级。

这是把世界从“机制游乐场”推进到“真实生态系统”的关键要求。

## R-007 · 训练、试炼、试运行、生产必须连成自动闭环，不能长期依赖人工搬运 checkpoint

### 用户要求

用户不希望项目长期停留在“训练完手动跑测试，测试完再手动放生产”的割裂流程。用户希望这几部分不是互相断开的场景，而是**同一谱系/种群连续流转的闭环系统**。

生产世界里暴露出来、且无法靠系统内短期自然进化自行修复的问题，应能自动回流到前端，转化为新的训练挑战、筛选条件和晋升依据。

### 核心要求

- 默认流程应是：`孵化 -> 试炼 -> 边境试运行 -> 正式世界 -> 问题回流 -> 再孵化`
- 进入正式世界的不是孤立模型文件，而是可追踪的 `lineage / population`
- 生产 incident 必须能沉淀为 challenge，而不是只留一堆日志
- 闭环默认自动运行，人工干预应是例外，不应是主流程

### 稳定性要求

- 不允许新谱系直接全量替换正式世界
- 必须保留稳定基线与历史 archive，避免单一路线崩盘
- 新谱系必须先影子运行或小流量放归，再决定是否扩大比例
- live 表现恶化时必须能自动降权、撤回或回滚
- 任何晋升都不能只看单个总分，必须同时看基础生存、专项修复、历史回归和风险信号

### 动态调控要求

用户希望闭环中的关键接口尽量不是全靠人工写死规则，而是可由神经网络/学习型控制器动态调控。允许存在：

- `incident -> challenge` 的动态映射器
- curriculum / task scheduler
- archive / lineage 调度器
- promotion scorer / risk estimator

但这些学习型控制器只能负责**提议、排序、加权、调度**，不能无护栏地独占最终裁决。

### 硬护栏要求

即使引入学习型控制器，系统仍必须保留至少以下硬门槛：

- 不能显著差于稳定基线
- 必须通过历史回归集
- 必须通过小流量试运行
- 出现恶化时必须允许自动回滚

也就是说，闭环可以高度神经化，但不能完全取消保底规则。

### 实现方向参考

- Safe Policy Improvement：保守晋升，先保证不比基线差
- Canary / Shadow Deployment：先影子运行、小流量放归、再扩大
- Population Based Training：异步种群筛选与持续晋升
- Quality Diversity / MAP-Elites：保留多样稳定 archive，避免单路进化自毁
- Prioritized Level Replay / Reverse Curriculum / UED / Adaptive Task Scheduler：把生产问题动态转成训练挑战与课程

### 文献支持

- Scholl et al., 2022, `Safe Policy Improvement Approaches and their Limitations`
- Jaderberg et al., 2017, `Population Based Training of Neural Networks`
- Pugh et al., 2016, `Quality Diversity: A New Frontier for Evolutionary Computation`
- Florensa et al., 2017, `Reverse Curriculum Generation for Reinforcement Learning`
- Jiang et al., 2021, `Prioritized Level Replay`
- Yao et al., 2021, `Meta-learning with an Adaptive Task Scheduler`

### 必须避免的错误实现

- 把 train/test/prod 做成三个长期割裂的手动菜单
- 新模型一次性替换整个正式世界
- 把所有判断都交给单个神经网络黑箱拍板
- 只有 incident 日志，没有 challenge 回流
- 只有自动前进，没有自动回滚

### 实现优先级

高优先级。

这是让项目从“离线训练项目”升级为“会自我纠错、自我育种、自我放归的长期生态系统”的关键要求。

## R-008 · 系统应允许网络在更丰富的数据与环境压力下自长技能、分工与局部机制，而不是什么都人工写死

### 用户要求

用户希望项目后续不是所有行为机制都只能靠人工先写死，再让模型去适配；系统应尽量允许网络在更丰富的感知、记忆、任务压力和生态互动中，**自己长出新的技能、行为分化、局部模块化甚至通信/协作模式**。

### 核心要求

- 不能把网络永久锁死成只有固定套路的黑箱
- 应给网络留下结构增长、行为分化、模块分工和技能沉淀的空间
- richer data 本身不够，必须同时有选择压力、任务差异、环境变化和多样性保留
- 新长出的能力不能一出现就被旧主流策略压死

### 实现方向参考

- NEAT / HyperNEAT：允许拓扑复杂化，给“长机制”留结构空间
- Option-Critic：允许时序技能自己分化
- DIAYN：允许无监督技能簇自己长出来
- RIMs：鼓励内部模块分工而不是所有功能混成一团
- PAIRED / UED：让环境自动出更合适的新题，逼出新行为

### 文献支持

- Stanley & Miikkulainen, 2002, `Evolving Neural Networks through Augmenting Topologies`
- Bacon et al., 2017, `The Option-Critic Architecture`
- Eysenbach et al., 2018, `Diversity is All You Need: Learning Skills without a Reward Function`
- Goyal et al., 2019, `Recurrent Independent Mechanisms`
- Dennis et al., 2020, `Emergent Complexity and Zero-shot Transfer via Unsupervised Environment Design`

### 必须避免的错误实现

- 只加更多输入维度，却不给新的选择压力
- 只追单一总分，导致刚长出的新行为立刻被淘汰
- 结构上允许增长，但没有 archive / speciation / 多样性保留
- 把“会自动长机制”理解成完全不需要外部设计世界和训练压力

### 实现优先级

中高优先级。

这是项目从“会拟合既定任务”走向“会在世界中自己长出新能力”的关键要求。
