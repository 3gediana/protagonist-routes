# 调试方法论总览（protagonist_ecology）

> 写给接手的模型：本目录每个 `0X_*.md` 是一个真实案例。PE 和 DP 共用同一套"先量再改、用预判器分钟级读信号、改了没动也是证据"的方法论（DP 篇 `00` 有更完整的展开，建议也读一遍），这里只讲 PE 特有的部分。

---

## 0. PE 是什么（一句话）

NEAT 演化：一个 24 个体的 protagonist 种群，按 7 个 harmony（和谐）指标做协同演化。关键观察量是**振幅 amplitude**（行为/生态指标随代数的振荡幅度），目标是突破 ~0.009 的天花板。

## 1. PE 特有的两个坑

### 坑 A：度量盲点——"指标根本没在你以为的时候写盘"
PE 的 trace/振幅日志**只在每个长跑的最后 20 代（`tail_generations=20`，即 gen140–160）才写盘**。后果：
- 一个从 gen80 热 resume 的长跑，前 60 代**完全读不到振幅**，你盯着日志会以为"什么都没发生"。
- 想做"分钟级"读数的实验，如果实验只跑到 gen100，就**永远拿不到振幅**——白烧 CPU。

**教训（PE 第一定律）：设计任何实验前，先搞清楚你要的指标在代码里到底是何时、何地写盘的。** 详见 `03_度量盲点.md`。

### 坑 B：预判器的"迁移有效性"边界
PE 的早代预判器是在 **sC 冷启动**（gen0–15 的特征 → 预测 gen79 的结果）上**标定并验证**的。把它直接套到"从 gen80 热 resume 的长跑"属于**未验证的迁移**——分布不同，贸然用它开新短跑可能得出错误方向。

**教训：预判器只在它被标定过的工况内可信。换工况（冷启动→热 resume）必须重新验证，否则它给的方向是噪声。** 详见 `02_早代预判器.md`。

## 2. PE 与 DP 资源分工

- DP 吃 GPU（PPO/libtorch），PE 吃 CPU（NEAT 演化），天然不抢资源 → **两个项目可同时跑、榨干 A10**。
- PE 多 seed 并行铺满核心（如 3 个 seed × CPU 多核）。

## 3. 当前状态快照

- 3 个 sD_warm seed 在跑（目标 gen160），CPU 多核吃满，正常演化。
- 振幅只在 tail（gen140+）可读 → 这几个长跑给不了分钟级读数，要等它们推进到 tail 才拿真振幅。
- 振幅历史：在 0.004–0.008 间徘徊，**尚未突破 0.009 天花板**（见 `01_振幅平台.md`）。

## 4. 一句话给接手模型

**PE 的两条铁律：① 动手前先确认指标的写盘时机（tail_generations=20，前期读不到振幅）；② 预判器只在标定过的工况内可信，换工况要重新验证。其余沿用 DP 的方法论：先量再改、每次只动一个变量、改了没动=排除假设。**
